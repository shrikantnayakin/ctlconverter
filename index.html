Before You Begin

This guide will take you from zero to a working barcode CTL file — even if you have never seen one before. Read this page first.

What is a Barcode Label?

A barcode label is the small sticker attached to every garment or product in a Ginesys store. It contains the item's price, category, barcode number, and other details. When a cashier scans a product at the POS counter, the barcode on that label is what drives the transaction.

Every item in Ginesys must be barcoded before it can be sold. The barcode is also used for stock movements, returns, and inventory counts.

What is a CTL File?

A CTL (Control) file is a plain text file that acts as a template for printing barcode labels. Think of it as a "fill-in-the-blank" form: it contains all the fixed printer instructions plus special placeholders like #1BARCODE# or #1ACTMRP#. When Ginesys prints labels, it reads the CTL file and replaces each placeholder with the real data for each item.

📄
PRN File
The raw printer file from the label design software. Contains fixed sample data. This is what the customer gives you.
⚙️
CTL File
The converted template. Sample data is replaced with #FIELD# tags. This is what you upload into Ginesys.
🖨️
Printed Label
Ginesys merges real item data with the CTL template and sends the result to the printer.

How PRN → CTL Conversion Works

Understanding what actually happens during conversion will help you debug problems and create CTL files manually when needed.

The Three Things That Change

Converting a PRN to a CTL involves exactly three types of change. Everything else in the file stays the same:

#What changesExample (before)Example (after)
1Add a UPS header on line 1(nothing — PRN has no UPS line)UPS2
2Wrap each data line in apostrophes (HO only)1911C0803300040KP-BRAND'1911C0803300040KP-BRAND'
3Replace sample data values with #FIELD# tags'1911C0803300040KP-BRAND''1911C0803300040'||#1CNAME1#||''

Side-by-Side: PRN vs CTL

Here is a real example showing a PRN and the two CTL variants generated from it:

1911C1003290252KP-DESCRIPTION

'1911C1003290252'||#1ARTICLENAME#||''

What "Mapping" Means

The PRN contains sample data — values that the customer used when designing their label, like KP-BARCODE-123, KP-BRAND, or 199.00. These are not real item data; they are just placeholders used for design.

When you "map" a field, you are telling the tool: "Wherever you see this sample value in the PRN, replace it with this Ginesys field tag." The tag then gets filled in with real data from the Ginesys database at print time.

⚠️

Not every line in a PRN contains customer data. Lines like ^Q20,3, ^W80, ^L, or Dy2-me-dd are printer configuration commands. These must be left exactly as-is (just wrapped in apostrophes for HO). Do not replace them with field tags.

Oracle vs PostgreSQL

If you have chosen HO, you need to know which flavour of HO the customer is using. In most deployments this is Oracle. Newer Ginesys installations may be on PostgreSQL.

How to Tell Which One

Ask the customer or the Ginesys support team. They can check in the Ginesys configuration. Alternatively, if you have an existing working CTL file for this customer, look for these clues:

If you see this in an existing CTL…It is…

COALESCE(, CURRENT_DATE, TO_TIMESTAMP(PostgreSQL

Step-by-Step Guide

Creating an HO CTL File

A complete walkthrough for creating an HO (Head Office) CTL file using the converter tool. Use this when the customer prints labels from GinesysWeb.

Step 1 — Get the PRN file

Ask the customer for their .prn file. This is the raw print layout exported from their label design software (Bartender, NiceLabel, or the printer's native software). Do not accept a screenshot or photo — you need the actual file.

Open the PRN file in Notepad to preview it. You will see lines of printer commands mixed with sample data values. Do not edit anything yet.

Step 2 — Set database and module in the tool

Click the 🗄 Oracle › Item ✎ button. Select Oracle (or PostgreSQL) and the Item module. This loads the correct list of Ginesys field tags for the dropdowns.

Step 3 — Open the PRN in the tool

Click 📂 Open PRN File and select the file. The tool will:

  • Detect the printer format (Argox, Citizen, or Zebra ZPL)
  • Extract all unique data values from the file
  • Auto-detect the UPS count (how many labels per row)
  • Show the original PRN on the right panel

Step 4 — Confirm UPS with the customer

Look at the UPS badge in the file bar. Check whether this matches the physical label sheet. For example, if the sheet has 2 labels side-by-side, it should say UPS2. If it's wrong, the labels will print with offset data.

Step 5 — Map each field

The left panel shows every unique data value found in the PRN. For each row, select the Ginesys field tag that matches. Common mappings:

PRN value (example)What it likely isGinesys tag
KP-BARCODE-123 or any barcodeBarcode number#1BARCODE#
KP-BRAND, POLO, brand namesCategory 1 (Brand)#1CNAME1#
RED, BLUE, colour valuesCategory 3 (Colour)#1CNAME3#
S, M, L, XLCategory 4 (Size)#1CNAME4#
199.00, 499, price valuesActual MRP#1ACTMRP#
KP-DESCRIPTION, item nameArticle name#1ARTICLENAME#
MENSWEAR, LADIESDepartment#1DEPARTMENT#

Use ⚡ Auto Map first, then review. The CTL Preview tab on the right updates live as you map.

Step 6 — Handle price fields with SQL functions

For price fields (MRP, RSP), using a plain tag like #1ACTMRP# sometimes shows too many decimals or causes errors if the value is empty. Use the function picker (ƒ in the dropdown) to apply a safe formatting function. See HO SQL Functions for details.

Step 7 — Generate and validate

Click ✨ Generate CTL. A file downloads. Then open 🚀 Migration → Validate CTL, drop the downloaded file in, and check for any errors before delivering to the customer.

Step-by-Step Guide

Creating a POS CTL File

A complete walkthrough for POS (store-level) CTL files. Use this when the customer prints labels from the POS Back Office.

Step 1 — Get the PRN file

Same as HO — obtain the .prn file from the customer and open it in Notepad to preview the contents before starting.

Step 2 — Set database to POS in the tool

Click the 🗄 Oracle › Item ✎ button. Select POS as the database, then Item as the module. The field tag list will switch to POS-specific tags (e.g. #1RSP# instead of #1ACTMRP#).

Step 3 — Open the PRN and map fields

The same process as HO — open the file, use Auto Map, then review each row. The key difference is in the field tags you use:

For this data…Use this tag in POSNOT this HO tag
Retail selling price (MRP)#1RSP##1ACTMRP#
Category attributes#1CAT1#, #1CAT2#, …#1CNAME1#
Item description#1DESCRIPTION##1ARTICLENAME#

Step 4 — Handle price fields safely

For POS price fields, always use the safe CASE WHEN pattern to guard against NULL and blank values. Open the function picker and select Full safe price pattern under the ISNULL / CASE category. The tool applies it automatically.

POS — Safe price line (RSP)
<F>'AC,97,51,1,1,0,0,'+ Case WHEN '#1RSP#' = '' THEN '0'
  WHEN ISNULL('#1RSP#', '2') = '0' THEN '0'
  ELSE convert(varchar(20),convert(numeric(20,0),'#1RSP#'))
  END +''

Step 5 — Generate and test

Generate the CTL, validate it, then test it on the actual POS printer with real item data before delivering to the customer. POS labels often have subtle SQL Server quirks that only show with live data.

Where to Upload the CTL in POS

POS Back Office → Store Control → Barcode Printing → browse for the CTL file.

Step-by-Step Guide

Manual Conversion — Without the Tool

If the tool is not available, you can convert a PRN file to a CTL manually using only Notepad. This section gives you everything you need to do it correctly for each printer format.

ℹ️

Manual conversion requires more care and is slower, but uses exactly the same rules as the tool. The tool simply automates these steps. Understanding them helps you debug problems even when using the tool.

Universal Rules (apply to every manual conversion)

RuleDo this
Line 1 must be the UPS declarationAdd UPS1, UPS2, or UPS3 as the very first line. Delete any blank lines before it.
Count labels per row correctlyLook at the physical label sheet. 1 across = UPS1, 2 across = UPS2, 3 across = UPS3.
Use UPPERCASE field names#1BARCODE# not #1barcode#
Do not modify config linesLines like ^Q20,3, ^W80, ^L, E are printer config — wrap in quotes but change nothing else.

Method 1 — Argox / Access Printer (HO Oracle)

Argox PRN files have data lines that start with a hex-decimal prefix like 1911S... or 1e.... The data value is the trailing suffix after the prefix.

Argox / Access — Manual conversion steps
1. Open PRN in Notepad. Add UPS line at top.

2. Wrap every line in apostrophes:
   BEFORE: 1911C0803460073KP-JS-BARCODE123
   AFTER:  '1911C0803460073KP-JS-BARCODE123'

3. Split each data line at the prefix boundary.
   Prefix = everything up to (but not including) the sample data
   Data   = the trailing sample value

4. Replace the data with a field tag using || concat:
   BEFORE: '1911C0803460073KP-JS-BARCODE123'
   AFTER:  '1911C0803460073'||#1BARCODE#||''

5. For static lines (config, literals), leave as-is:
   '^Q20,3'   ← no change, just quoted
   'E'        ← no change

Method 2 — Citizen POS Label Commands (AB / BP / AC / AT)

These PRN files use plain comma-separated commands. The data is always the last comma-separated field.

Citizen POS commands — Manual conversion (HO Oracle)
PRN line format: COMMAND,x,y,...,DATA

1. Add UPS line at top.

2. For each data line, identify prefix and data:
   AB,16,7,1,1,0,0,DEPARTMENT
   ├── prefix: AB,16,7,1,1,0,0,
   └── data:   DEPARTMENT

3a. HO Oracle — wrap in apostrophes + || concat:
   BEFORE: AB,16,7,1,1,0,0,DEPARTMENT
   AFTER:  'AB,16,7,1,1,0,0,'||#1DEPARTMENT#||''

3b. POS SQL Server — plain, no apostrophes:
   BEFORE: AB,16,7,1,1,0,0,DEPARTMENT
   AFTER:  AB,16,7,1,1,0,0,#1DEPARTMENT#

4. For static literal lines (e.g. MRP : text), HO wraps in quotes:
   BEFORE: AC,15,51,1,1,0,0,MRP :
   HO:     'AC,15,51,1,1,0,0,MRP :'       ← quoted, unchanged
   POS:    AC,15,51,1,1,0,0,MRP :         ← unchanged

5. For price lines with SQL (POS only):
   BEFORE: AC,97,51,1,1,0,0,19999.00
   AFTER:  <F>'AC,97,51,1,1,0,0,'+ Case WHEN '#1RSP#'='' THEN '0'
              WHEN ISNULL('#1RSP#','2')='0' THEN '0'
              ELSE convert(varchar(20),convert(numeric(20,0),'#1RSP#'))
              END +''

Method 3 — Zebra ZPL (HO Oracle)

ZPL files use ^FD and ^FS to delimit data fields. Split at these markers.

Zebra ZPL — Manual conversion steps
1. Add UPS line at top.

2. For each ^FD...^FS line, identify three parts:
   ^FT80,388^A0N,28,28^FH\^CI28^FDBRD:RICH BDR^FS^CI27
   ├── prefix:  ^FT80,388^A0N,28,28^FH\^CI28^FD
   ├── data:    BRD:RICH BDR
   └── postfix: ^FS^CI27

3. Build the CTL line (HO Oracle):
   BEFORE: ^FT80,388^A0N,28,28^FH\^CI28^FDBRD:RICH BDR^FS^CI27
   AFTER:  '^FT80,388^A0N,28,28^FH\^CI28^FD'||#1CNAME3#||'^FS^CI27'

4. If data has a literal prefix (e.g. Rs.), fold it into the prefix:
   BEFORE: ^FT80,426^A0N^FDRs.199.00^FS^CI27
   AFTER:  '^FT80,426^A0N^FDRs.'||#1ACTMRP#||'^FS^CI27'  ← Rs. stays

5. Static config lines — just wrap in quotes:
   BEFORE: ^XA
   AFTER:  '^XA'
💡

Tip for UPS2+ labels: The PRN will contain each data element twice (once per label position). The first occurrence gets #1FIELD#, the second gets #2FIELD#. If a field appears 3 times, use #1, #2, #3.

Field Tags Explained

Field tags are the core of a CTL file. They are the placeholders that Ginesys replaces with real item data at print time.

Anatomy of a Field Tag

Field tag breakdown
#  1  ACTMRP  #
│  │  │        └── closing hash — marks end of tag
│  │  └─────────── field name — UPPERCASE, matches Ginesys master column
│  └────────────── position number — 1 = first label, 2 = second, 3 = third
└──────────────── opening hash — marks start of tag

Rules for Field Tags

RuleCorrectWrong
Must be UPPERCASE#1BARCODE##1barcode#
Must have a position number#1ACTMRP##ACTMRP#
Must have opening and closing ##1BARCODE##1BARCODE
No spaces inside the tag#1CNAME1##1 CNAME 1#

In HO CTL — How a Tag Appears in a Line

In HO mode, the tag sits between apostrophe-quoted sections joined by || (pipe-pipe) concatenation:

HO Oracle — apostrophe + concat pattern
Plain field — no prefix/suffix:
'1911C0803460073'||#1BARCODE#||''
 ↑ prefix in quotes     ↑ tag    ↑ empty suffix (still required)

With literal prefix (e.g. Rs.):
'1911C1202230040Rs.'||#1ACTMRP#||''

With literal suffix (e.g. .00):
'1911C1202230040'||#1ACTMRP#||'.00'

In POS CTL — How a Tag Appears in a Line

In POS mode, the tag is inserted directly into the line — no apostrophes or concat operators for plain fields:

POS — plain field insertion
Plain field — tag replaces the data value directly:
AB,16,7,1,1,0,0,#1DEPARTMENT#
BP,31,85,2,6,45,0,0,#1BARCODE#

When a SQL function is needed, use <F> prefix:
<F>'AC,97,51,1,1,0,0,'+ SUBSTRING('#1DEPARTMENT#',1,25) +''

The UPS System

UPS stands for "Units Per Strip" — it tells Ginesys how many labels are printed side-by-side on each row of the label sheet. Getting this right is essential for correct data alignment.

What UPS Means in Practice

🏷️
UPS1
1 label per row. Every field uses #1FIELD#. Most common for large labels.
🏷️🏷️
UPS2
2 labels side-by-side per row. Left label uses #1FIELD#, right label uses #2FIELD#.
🏷️🏷️🏷️
UPS3
3 labels side-by-side per row. Use #1, #2, #3 prefixes for each position.

How to Count UPS

Look at the PRN file. Find a data field that you know represents the barcode or a unique per-item value. Count how many times it appears in the file. That count is your UPS value.

For example, if the sample barcode KP-BARCODE-123 appears 2 times in the PRN, the UPS is 2. The tool auto-detects this and shows it in the file bar.

UPS2 Example — How the Lines Pair Up

PRN (UPS2 — each element appears twice)
1911C0803460073KP-BARCODE-123    ← barcode, label 1
1911C0803460073KP-BARCODE-123    ← barcode, label 2 (same sample value)
1911C0803300040KP-BRAND          ← brand, label 1
1911C0803300040KP-BRAND          ← brand, label 2
CTL (HO Oracle UPS2)
UPS2
'1911C0803460073'||#1BARCODE#||''    ← label 1 barcode
'1911C0803460073'||#2BARCODE#||''    ← label 2 barcode
'1911C0803300040'||#1CNAME1#||''     ← label 1 brand
'1911C0803300040'||#2CNAME1#||''     ← label 2 brand
⚠️

All labels show the same data? This means you used #1FIELD# for every line instead of alternating #1 and #2. For UPS2, the odd-numbered lines use #1 and the even-numbered lines use #2.

SQL Functions

POS SQL Functions — The <F> Syntax

In POS CTL, any row that uses a SQL Server function must start with <F>. This tells the POS engine to evaluate the expression as T-SQL before printing. Rows without <F> are treated as plain text.

When Do You Need <F>?

SituationNeed <F>?Example
Plain field tag, no transformationNoAB,16,7,1,1,0,0,#1DEPARTMENT#
Static literal text onlyNoAC,15,51,1,1,0,0,MRP :
Field needs trimming, formatting, or length limitYes<F>'...'+ SUBSTRING('#1DEPT#',1,25)
Price needs NULL/blank safety checkYes<F>'...'+ CASE WHEN '#1RSP#'=''...
Date needs formattingYes<F>'...'+ convert(varchar(10),GETDATE(),103)
Today's date (GETDATE)Yes<F>convert(varchar(10),GETDATE(),103)

Syntax Pattern

POS <F> syntax pattern
<F>'static-prefix'+ SQL_FUNCTION('#1FIELD#') +''
 ↑    ↑                  ↑                  ↑               ↑
 │    │                  │                  │               └ empty closing literal
 │    │                  │                  └─────────────── field tag in SINGLE QUOTES
 │    │                  └────────────────────────────────── T-SQL function
 │    └───────────────────────────────────────────────────── static text (the printer prefix)
 └────────────────────────────────────────────────────────── required prefix — signals SQL eval

Printer Formats

Argox / Access Printer Format

Argox and Access brand label printers use a hex-decimal prefix to encode position and print parameters. The label data is the plain text suffix that follows the prefix on each line.

Recognising This Format

Look at the PRN file in Notepad. If lines start with 1911S, 1911C, 1911B, or 1e followed by many hex digits, it is Argox/Access format.

Argox PRN — example lines
1911S0803460073KP-BARCODE-123      ← text line: prefix + data
1e6302803580040B                    ← barcode element (data in following line)
1911C0803300040KP-BRAND
~Q+0                                ← printer config — leave unchanged
^L                                  ← printer config — leave unchanged

How to Identify the Prefix and Data

The regex the tool uses is ^(1[0-9a-fA-F]{2,3}[A-Za-z][0-9a-fA-F]{4,}(?:[A-Z][0-9a-fA-F]{3})+)(.+)$. In plain terms: the prefix is the hex instruction block, and the data is everything after it.

CTL Output

Argox CTL (HO Oracle)
UPS1
'~Q+0'                                         ; config — quoted as-is
'^L'                                           ; config — quoted as-is
'1e6302803580040B'||#1BARCODE#||''          ; barcode graphic
'1911C0803460073'||#1BARCODE#||''           ; barcode text
'1911C0803300040'||#1CNAME1#||''            ; brand
'1911C1202230040Rs.'||TO_CHAR(NVL(#1ACTMRP#,0),'FM99990.00')||''

Citizen Printer Format

Citizen label printers use two distinct PRN formats. The classic 1911C format has a decimal-encoded prefix. The newer POS label command format uses plain commands like AB, BP, AC, AT with data as the last comma-separated value.

Format 1 — Classic Citizen (1911C)

Lines start with 1911 followed by a letter and exactly 10 decimal digits. The data is everything after those 15 characters.

Classic 1911C PRN — anatomy
1911C0803460073KP-BARCODE-123
│    │││││││││└──── data value
│    ││└──────────── 10 decimal coordinate digits
│    │└───────────── command letter
└────┴────────────── Citizen prefix

Format 2 — Citizen POS Label Commands

Used in store POS label printers. Commands like AB (text), BP (barcode), AC (small text), AT (large/bold text). The data is always the last comma-separated field.

Citizen POS command PRN — anatomy
AB,16,7,1,1,0,0,DEPARTMENT
│  │ │         └─── data value (last comma field)
│  └─┴─────────── x, y coordinates + parameters
└──────────────── command letter(s)

Printer Formats

Zebra ZPL Printer Format

Zebra printers use ZPL (Zebra Programming Language). Every label is defined between ^XA and ^XZ markers. Data fields sit between ^FD (field data start) and ^FS (field end).

Recognising This Format

If the PRN contains lines with ^FD and ^FS, it is Zebra ZPL format. Config commands start with ^ (caret).

ZPL PRN — anatomy
^FT80,388^A0N,28,28^FH\^CI28^FDBRD:RICH BDR^FS^CI27
│                              │  ├────────────┤  └── restore charset
│                              │  data field       field end
│                              └── field data start (^FD)
└── position + font parameters

How to Convert ZPL Lines Manually

Split each ^FD...^FS line into three parts: everything up to and including ^FD, the data value, and everything from ^FS onwards. Wrap in quotes and join with || concat:

ZPL — Manual conversion (HO Oracle)
PRN:  ^FT80,388^A0N,28,28^FH\^CI28^FDBRD:RICH BDR^FS^CI27

Split:
  prefix  = ^FT80,388^A0N,28,28^FH\^CI28^FD
  data    = BRD:RICH BDR
  postfix = ^FS^CI27

CTL:
'^FT80,388^A0N,28,28^FH\^CI28^FD'||#1CNAME3#||'^FS^CI27'

With literal prefix (Rs.) already in the data field:
  PRN data = Rs.199.00  →  merge Rs. into prefix:
'^FT80,426^A0N^FDRs.'||#1ACTMRP#||'^FS^CI27'

CTL Syntax Rules

Universal Rules

RuleDetail
Line 1 always UPSThe very first line must be UPS1, UPS2, or UPS3 — nothing before it, including blank lines.
Field tag syntaxAll replaceable field names enclosed in #: #1FIELDNAME#
Numeric prefix1 = first label in row, 2 = second, 3 = third (matches UPS count)
Case sensitivityField names must be UPPERCASE: #1BARCODE# not #1barcode#

HO CTL Rules (Oracle & PostgreSQL)

RuleDetail
Apostrophe wrappingEvery data line must start and end with a single apostrophe '
Field tag concatUse || for concat: 'prefix'||#1FIELD#||'suffix'
Oracle functionsSUBSTR, NVL, TO_CHAR, DECODE, SYSDATE, TRUNC
PostgreSQL functionsSUBSTR, COALESCE, TO_CHAR, CAST, CURRENT_DATE, DATE_TRUNC

POS CTL Rules (SQL Server)

RuleDetail
<F> prefixRequired on any row that uses SQL Server function evaluation
No apostrophe wrappingPlain field lines do not use apostrophes
T-SQL functionsSUBSTRING, ISNULL, CONVERT, GETDATE, CAST, LEN, LEFT
+ concatenationUse + (not ||) in <F> rows
Field tags in SQLInside SQL functions, wrap field tags in single quotes: '#1RSP#'
Date format codes103 = DD/MM/YYYY · 105 = DD-MM-YYYY · 120 = YYYY-MM-DD

Zebra ZPL Rules

RuleDetail
Field delimiter splitData sits between ^FD and ^FS. Prefix ends with ^FD, postfix starts with ^FS.
CTL line pattern'<prefix>^FD'||#1FIELD#||'^FS<postfix>'
Literal prefix inside fieldMerge it into the prefix literal: '…^FDRs.'||#1ACTMRP#||'^FS^CI27'

Troubleshooting

ProblemLikely causeFix
No fields appear in the mapping table after opening PRN Printer format not recognised, or file is empty Check the PRN: supported formats are Argox/Access (1911S…), classic Citizen (1911C + 10 digits), Citizen POS label commands (AB/BP/AC/AT), and Zebra ZPL (^FD…^FS). Use ✏️ Edit Existing CTL or manual conversion for unsupported formats.
Label prints blank (no data) Field tag does not match the database master mapping Confirm exact field name with the customer's item master. Check spelling of #1FIELD#. Run Validate CTL to catch malformed tags.
SQL function not evaluated in POS Missing <F> prefix on the row Add <F> at the start of the row. The Validator flags this automatically.
Wrong number of labels per row UPS value doesn't match the physical label sheet Count labels on the sheet. Set UPS to match. Auto-detection uses field repetition frequency — verify it in the file bar after loading.
All labels show the same item's data Used #1FIELD# for all positions instead of #1, #2, #3 For UPS2, alternate between #1FIELD# (odd lines) and #2FIELD# (even lines). The tool does this automatically.
Price shows 0 or NULL Price field is null or empty in the database Use the safe CASE WHEN pattern from the function picker. See POS SQL Functions.
MRP prints as Rs.35958.00.00 Hardcoded .00 suffix + DB value already includes decimals Remove the hardcoded '.00' suffix from the CTL line if the database field already returns decimals.
Date format is wrong Wrong CONVERT format code (POS) or wrong TO_CHAR mask (HO) POS: use code 103 for DD/MM/YYYY, 105 for DD-MM-YYYY. HO: use 'DD/MM/YYYY' mask in TO_CHAR.
Garbled text on label PRN file encoding mismatch or wrong codepage Check the CODEPAGE or ^CI setting. For Zebra, ^CI28 = UTF-8.
Validate shows "Oracle function in PostgreSQL CTL" CTL was originally written for Oracle Use 🚀 Migration → Oracle → PostgreSQL to auto-convert all functions.
Custom mapping rules / presets lost after closing Using v1 of the tool (no localStorage persistence) Upgrade to v2. All data now persists automatically via localStorage.
HO CTL preview shows no substitution for Citizen POS label PRN Using v1 — Citizen handler only ran in POS mode Upgrade to v2. The Citizen CMD handler now runs for all database modes.

Release Notes

What's New in v2

Version 2 brings persistent data storage, full Citizen POS label printer support for all database modes, intelligent function auto-substitution, a fully restructured Knowledge Base, and several reliability fixes.

1 — Data Persists Across Sessions & Systems

In v1, all user data (custom mapping rules, function templates, presets, user accounts) was lost on every page reload. In v2, everything is automatically saved to localStorage and survives reloads and browser restarts. To move data to another PC, use ⬇ Export Module / ⬆ Import Module in the Edit Mapping editor.

2 — Citizen POS Label Format Now Works for HO Mode

PRN files using AB, BP, AC, AT commands are now fully supported for both POS and HO (Oracle/PostgreSQL) database modes. UPS detection also fixed — same barcode in BP + AT commands no longer inflates the count.

3 — Function Templates Auto-Populate with the Mapped Field

Templates with hardcoded field names (e.g. SUBSTR(#1ARTICLECODE#,1,8)) now auto-replace the field with whatever you have mapped in that row. The function picker list shows a cyan preview of the resolved expression before you click. The CTL Output Preview at the bottom of the builder now shows the correct format (HO vs POS).

4 — Knowledge Base Restructured for Beginners

The KB is fully reorganised from a reference manual into a learning path: Getting Started → Choose Your Database → Step-by-Step Guides → Field Mapping → SQL Functions → Printer Formats → Tool Reference → Reference. Manual conversion instructions (without the tool) are included for every printer format.
